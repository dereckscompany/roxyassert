
# roxyassert

> Write the contract once тАФ in your roxygen2 docs тАФ and generate the
> runtime checks.

`roxyassert` reads **structured type annotations** in your roxygen2
documentation and, at `document()` time, **generates per-function
assertion helpers** that validate a functionтАЩs arguments and its return
value. The generated checks are plain calls to the
[`assert`](https://github.com/dereckscompany/assert) package, written to
a committed source file you can read and debug.

The point: you already describe each parameterтАЩs type and each return
shape in your `@param` / `@return` docs. Today you *also* hand-write a
separate validation function that repeats all of it тАФ and the two drift.
`roxyassert` makes the documentation the single source of truth: the
docs render for humans **and** generate the assertions, so they can
never disagree.

## How it works

      R/*.R  (your code + typed roxygen tags)
         |
         |   devtools::document()   <- roxyassert runs here, as a roclet
         v
      R/contracts-generated.R       <- generated assert_args_* / assert_return_* helpers
         |
         |   you call them inside your functions, and commit the file
         v
      runtime validation via the `assert` package

`roxyassert` is a **roxygen2 roclet**: it runs in the documentation step
(the same one that writes `NAMESPACE` and `man/*.Rd`), never during
`R CMD build`/`check`. It parses tag *text* only тАФ it never executes
your code тАФ and writes one file you commit alongside the rest of your
generated artifacts.

Layering:

| Layer | Package | Role |
|----|----|----|
| Vocabulary | [`assert`](https://github.com/dereckscompany/assert) | The runtime primitives (`assert_scalar_character()`, `assert_data_table()`, тАж). |
| Generator | **`roxyassert`** | Parses typed docs -\> emits helpers that *call* `assert`. |

`assert` has no dependency on `roxyassert` and stays lightweight;
`roxyassert` depends on `assert` + `roxygen2`.


## Setup (one line)

Register the roclet in your packageтАЩs `DESCRIPTION`:

``` dcf
Roxygen: list(markdown = TRUE, roclets = c("namespace", "rd", "roxyassert::contract_roclet"))
```

ThatтАЩs the entire integration. Now `devtools::document()` runs
`roxyassert` alongside the built-in roclets and (re)writes
`R/contracts-generated.R`.

## Quick example

``` r
#' Submit an order.
#'
#' @param symbol (scalar<character>) normalised `BASE/QUOTE` pair.
#' @param side (enum<character, ORDER_SIDE>) order side.
#' @param quantity (scalar<numeric>) order size in the base asset.
#' @param price_limit (scalar<numeric>?) limit price; `NULL` for market orders.
#' @return (data.table) the accepted order:
#' - **order_id** (character): exchange order id.
#' - **status** (character): order status.
#' - **quantity** (numeric): accepted size.
#' - **datetime** (POSIXct): acceptance time.
#' @export
submit_order <- function(symbol, side, quantity, price_limit = NULL) {
  assert_args_submit_order(symbol, side, quantity, price_limit)   # generated
  result <- ...
  return(assert_return_submit_order(result))                      # generated
}
```

After `document()`, `R/contracts-generated.R` contains (illustrative):

``` r
# Generated by roxyassert -- do not edit by hand.

assert_args_submit_order <- function(symbol, side, quantity, price_limit) {
  assert::assert_scalar_character(symbol)
  assert::assert_value_in_set(side, ORDER_SIDE)
  assert::assert_scalar_numeric(quantity)
  if (!is.null(price_limit)) assert::assert_scalar_numeric(price_limit)
  invisible(NULL)
}

assert_return_submit_order <- function(value) {
  assert::assert_data_table(value)
  assert::assert_column_types(value, "character", c("order_id", "status"))
  assert::assert_column_types(value, "numeric", "quantity")
  assert::assert_column_types(value, "POSIXct", "datetime")
  return(value)   # returned untouched, so you can wrap it in return()/invisible()
}
```

## The annotation grammar

A type annotation is a parenthesised token at the **start** of a
`@param` or `@return` description. Plain prose `@param`/`@return` tags
without a leading `(...)` token are ignored тАФ so adoption is incremental
and opt-in.

### Scalars and vectors

| Annotation                 | Meaning                              |
|----------------------------|--------------------------------------|
| `(scalar<character>)`      | length-1 character                   |
| `(scalar<numeric>)`        | length-1 numeric                     |
| `(vector<numeric>)`        | numeric of any length (\>= 1)        |
| `(vector<character>[10])`  | character of **exactly** 10 elements |
| `(vector<numeric>[1..10])` | numeric of length **1 to 10**        |
| `(vector<integer>[2..])`   | integer of length **at least 2**     |

`scalar<T>` is shorthand for `vector<T>[1]`.

### Base types

`character`, `numeric`, `integer`, `logical`, `complex`, `factor`,
`list`, `function`, `POSIXct`, `Date`.

### Modifiers

| Annotation | Meaning |
|----|----|
| `(scalar<numeric>?)` | **nullable / optional** тАФ `NULL` or a missing argument passes; otherwise the inner type is enforced. |
| `(enum<character, ORDER_SIDE>)` | value(s) must be members of the set/constant `ORDER_SIDE` (any object resolving to a vector of allowed values, e.g.┬аa `core` constant). |
| `(R6<Engine>)` | an R6 instance inheriting class `Engine`. Use this for builder methods that return `invisible(self)`. |

### data.tables / data.frames

Annotate the tag `(data.table)` (or `(data.frame)`), then declare the
columns as a **markdown bullet list** in this exact form:

``` r
#' @return (data.table) open orders:
#' - **order_id** (character): exchange order id.
#' - **price** (numeric): limit price.
#' - **datetime** (POSIXct): submission time.
```

Each bullet `- **name** (type): description` becomes a column assertion.
Column **type** uses the base types above. Because every column is
checked for an atomic type, **list-columns are rejected by
construction** тАФ a nested/list column fails its declared type.

> The same bullet syntax is the human-readable column documentation you
> already write, so the docs and the generated schema come from one
> place.

## Generated functions тАФ conventions

- **Two helpers, each optional.** `assert_args_<fn>` is emitted only if
  the function has at least one typed `@param`; `assert_return_<fn>`
  only if `@return` carries a parseable type. Functions with no typed
  tags get nothing.
- **Explicit arguments.** `assert_args_<fn>(arg1, arg2, ...)` takes the
  documented parameters by name, in declaration order тАФ you call it
  explicitly so the contract is visible at the call site. (If you change
  a functionтАЩs arguments, re-run `document()` and update the call.)
- **Returns are passed through.** `assert_return_<fn>(value)` validates
  and returns `value` unchanged, so you can write
  `return(assert_return_fn(x))` or keep your own `invisible()`.
- **`invisible()` is irrelevant.** Whether a function returns visibly or
  invisibly is a print concern, not a contract тАФ generation depends
  solely on a typed `@return`. A `@return invisible(self)` written as
  prose is skipped; annotate `@return (R6<Engine>)` to assert it.
- **R6 methods are scoped by class.** A method `submit` on class
  `Engine` generates `assert_args_Engine__submit` (double-underscore
  separator), so two classes sharing a method name тАФ or a method and a
  free function with the same name тАФ never collide.
- **Collision-safe.** If two generated names would ever clash,
  generation aborts with a clear error rather than silently overwriting.
- **Internal, undocumented, committed.** Generated helpers are not
  exported and carry no `.Rd`; they live in a single
  `R/contracts-generated.R` (with a do-not-edit banner) that you commit
  like `NAMESPACE`. Each package generates and uses its own.

## Supported targets

- Plain functions: yes
- R6 class methods: yes
- S4 / Reference classes: not yet.

## Configuration

Defaults need no config. Basic options (active roclets, output) live in
the `DESCRIPTION` `Roxygen:` field. Richer settings (generated-file
path, strictness) are read from an optional `roxyassert.yml` if present;
absent, sensible defaults apply.

## Why not put the type in the code instead?

Packages like [`typed`](https://github.com/moodymudskipper/typed)
declare types inside the function body with a custom operator. That
works and checks at runtime, but it changes how every function is
written. `roxyassert` keeps your R code exactly as it is and treats the
**documentation** you already write as the contract.
